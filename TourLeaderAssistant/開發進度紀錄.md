## 三、已知技術決策

| 決策 | 說明 | 理由 |
|------|------|------|
| SwiftData 不使用 `@Relationship` 管理 Team 子模型 | TourMember、Expense 等以 `teamID: UUID` 關聯 | 避免 SwiftData cascade delete 的 bug 風險，手動清理更可控 |
| `TourDocument` 保留 `fileURL` 欄位，新增 `resolvedURL` computed property | 不寫 migration，以 `teamID + fileName` 重組路徑 | iOS 沙盒路徑在 App 更新後 UUID 段會改變；computed property 方案向後相容、無遷移風險 |
| `PlaceHotel` 等 Supporting Types 以 JSON String 儲存 | `floorsAndHoursData: String` 搭配 computed property decode/encode | SwiftData 不支援複雜巢狀結構直接存儲 |
| `SupabaseManager` 作為單一上帝物件 | 暫不拆分主類別，以 Extension 分責（+Search、+Photos）| 功能仍在快速迭代中，過早拆分會增加維護成本 |
| `@Query` 全表載入暫緩重構 | 目前維持 in-memory filter 寫法 | 重構曾導致按鈕失效，待穩定後謹慎重試 |
| `withThrowingTaskGroup` 實作 timeout | DietaryParser AI 路徑最多等 5 秒 | Foundation Models 回應慢時，30 人團若無 timeout 可能等數分鐘 |
| `hasBirthdayOnTrip` O(1) 演算法 | 最多迭代 2 次（當年生日、次年生日）| 原逐日迴圈 30 天行程要跑 30 次 Calendar 操作 |
| `DateFormatter` 宣告為 `static let` | 避免在 View 渲染路徑或迴圈中重複建立 | `DateFormatter` 建立成本高，static 只建一次 |
| `TourMemberParser` 自行實作 ZIP + XML 解析 | 不依賴第三方套件處理 xlsx/docx | 減少外部依賴，包體積更小 |
| Keychain 存儲設備 UUID | `KeychainManager` | 比 UserDefaults 更安全，防止 App 重裝後 ID 改變 |
| xcconfig 分離敏感設定 | Supabase URL / anon key 不進 git | `.gitignore` 已加入 `*.xcconfig` |

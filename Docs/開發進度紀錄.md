## 七、Gemini 街頭智慧 (Street Smart) 優化建議

> 註：以下為 Gemini 基於實務經驗提出的「邏輯無誤但實務上容易踩坑」之重點優化建議，待後續排程處理。

### 1. 記憶體流失陷阱：`.onAppear { Task { ... } }` 改為 `.task`
- **現狀**：專案中約有 9 處使用 `.onAppear { Task { await load() } }`。
- **隱患**：若使用者提早退出畫面，背景 Task 不會自動停止，可能導致存取已釋放視圖的閃退或記憶體浪費。
- **建議**：全面改用 iOS 15 的 `.task { ... }`，系統會在畫面消失時自動 cancel 任務。

### 2. 畫面卡頓元兇：直接建立 `DateFormatter()`
- **現狀**：專案中約有 19 處在 `body` 或 computed property 中動態建立 `DateFormatter`。
- **隱患**：`DateFormatter` 初始化非常耗能，在 List 或 ScrollView 內會造成嚴重的掉幀 (Drop frames)。
- **建議**：改用 iOS 15 超輕量的 `date.formatted(.dateTime.month().day())`，或將 formatter 改為 `static let` 提取。

### 3. OCR 爆記憶體危機：PDF 轉圖片的 `autoreleasepool`
- **現狀**：`TourMemberParser.swift` 在處理 PDF 時，會將所有頁面轉為 `CGImage` 並存入一個大陣列中再一次性處理。
- **隱患**：若載入頁數較多 (例如 15+ 頁) 的 PDF，極易觸發 iOS 的 OOM (Out Of Memory) 導致 App 崩潰。
- **建議**：改為逐頁處理，並在每頁處理迴圈內使用 `autoreleasepool`，用完即刻釋放圖片記憶體。

### 4. 資料庫延遲顯示的保險：手動 `try? modelContext.save()`
- **現狀**：各類表單（如 `AddExpenseView`、`AddTourFundView` 等）在 `insert()` 後直接 `dismiss()`，無明確存檔。
- **隱患**：在涉及關係 (Relationships) 綁定時，SwiftData 有時不會立即發送 UI 更新通知，導致上一頁的列表沒立刻出現新資料。
- **建議**：在所有 `AddXxxView` 的 `save()` 函數最後，統一補上一行 `try? modelContext.save()` 作為保證。

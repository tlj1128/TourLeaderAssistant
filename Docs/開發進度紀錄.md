## 二、最新 Build 修改內容

詳見 `Changelog.md`。以下為 Build 15 重點摘要：

### Build 15 — 團員名單 OCR 升級

**iOS 26 `RecognizeDocumentsRequest`**（圖片 + PDF）
- 設計核心：1 Vision row = 1 RawTable row，cell 內 `\n` 保留不展開，由 mapper 透過 field type 自行決定怎麼吃多行 cell
- iOS 18–25 圖片 / PDF / 相機入口整個隱藏，只保留 xlsx / docx
- 新 `TourMemberStructuredPreviewView`（cell 多行 `lineLimit(nil) + fixedSize` 撐高）；舊 `TourMemberOCRMappingView` 整支砍

**新欄位類型**
- `.passportFull`：護照號碼+發照+效期 三合一單欄多行
- `.nationalID`：台灣身分證；首位數字推性別（1=M / 2=F）
- `.remarkEssential`：備註只留機位/房/餐/飲食/過敏 等必要資訊，地址電話自動丟掉

**autodetect**
- `looksLikeHeaderRow`：每 cell 看第一行是否為 ≤ 8 字 label，≥ 2 cell 像 label 整列視為表頭
- `detectField`：內容驅動，偵測護照三合一、英中合一、單行身分證、序號等

**修復**
- DocXMLParser bug：同一 cell 多 `<p>` 改用 `\n` 串接（之前段落黏一起）

**接手點**
- 中文姓名 OCR 準確度仍是痛點（spike 16 筆裡 6 筆有錯，preview 可改）
- Vision 偶爾會把第二行 header 跟第一筆團員 cell 合併 → autodetect 跳掉 member 1

## 六、Swift / SwiftUI 升級檢查（2026/04/29 起）

逐批掃描專案，找出過時做法 / iOS 18+ 新 API / iOS 26 only 機會。基線 iOS 18+，Xcode 26 / Swift 6.x。

### 圖示說明

- ❗ 官方明寫 deprecated（最高優先）
- 🔵 有更好新寫法（中等）
- 🟢 iOS 18 新 API 可用
- 🟣 iOS 26 only（需 `#available` 雙路徑）
- ⚪ 純優化（一人開發判斷做不做）

### Batch 1：Models（已掃，2026/04/29）

整體很乾淨，全部用 `@Model`、無 `ObservableObject` 殘留、`HotelSupportingTypes` 已標 `Sendable`。發現如下：

| # | 項目 | 嚴重 | 位置 | 決議 |
|---|---|---|---|---|
| 1 | JSON-as-String 反模式（`countryCodesData` / `receiptImagePathsData` / 4 個 hotel `*Data`） | 🔵 | `Team.swift:17`、`Expense.swift:23`、`PlaceHotel.swift:13–16` | ⏸ Phase 7 上架前處理，與 #2 綑綁（schema 變更，需 migration） |
| 2 | 導入 `VersionedSchema` | ⚪ | 全 SwiftData models | ⏸ 與 #1 綑綁，下次改 schema 那刻一起導入 |
| 3 | enum 補 `Sendable` 一致化 | ⚪ | `TeamStatus`、`PaymentMethod`、`DocumentCategory`、`DietaryScope`、`DietaryCategory` | ⏸ 待全專案掃完後與「Swift 6 strict concurrency 是否打開」一起決策 |

### Batch 2：Managers（已掃，2026/04/29）

整體現代化（`@Observable` 已用、async/await 普遍、`@MainActor` 標註合理）。最戰略性發現是現有 Vision OCR 用的是 iOS 26 之前 API，無法解析表格結構——已提升到 Phase 6 處理。

| # | 項目 | 嚴重 | 位置 | 決議 |
|---|---|---|---|---|
| 1 | Vision OCR 升級到 `RecognizeDocumentsRequest`（雙路徑：iOS 18 fallback + iOS 26 結構化解析）。確認此 API 為 iOS 26+ only，但**不綁機型**（不需 Apple Intelligence、不限 iPhone 15 Pro+），覆蓋面大於飲食 AI 那條 | 🟣 | `TourMemberParser.swift:213, 238` | ⬆️ 提升到 Phase 6 |
| 2a | `CalendarManager.requestAccess` callback → async（iOS 17+ 已有 `requestFullAccessToEvents()` async 版） | 🔵 | `CalendarManager.swift:16` | 🔗 與 strict concurrency 決策綁同一波 |
| 2b | `NetworkMonitor.pathUpdateHandler` 內 `DispatchQueue.main.async` → `Task { @MainActor in }`（或 class 標 `@MainActor`） | 🔵 | `NetworkMonitor.swift:14` | 🔗 同上 |
| 3 | 7 個 singleton class 改 `final class`（`SupabaseManager` / `BackupManager` / `PlacePhotoManager` / `ReceiptPhotoManager` / `CalendarManager` / `AppConfigManager` / `TipStore`），與現有 `NetworkMonitor` 一致化 | 🔵 | 多檔 | 🔗 同上 |
| 4 | Swift 6 strict concurrency 警告（如 `BackupManager` 內 `ModelContext` 跨 await） | ⚪ | `BackupManager.swift` | 🔗 holding box，全專案掃完一次決策 |

### Batch 3：Views（已掃，2026/04/29）

60 檔案，用 grep 模式掃 + 二輪深掃語意層級 pattern。**官方 deprecated API 完全為零**（`NavigationView` / `actionSheet` / `navigationBarItems` / `edgesIgnoringSafeArea` / `UIScreen.main` 全 0 命中）；`@Observable` migration 已完成、`NavigationStack` 全面使用、`onChange` 用 iOS 17+ 兩參數 form、`PhotosPicker` / `confirmationDialog` 都對。發現主要在 grep 才抓得到的「同義新寫法」：

| # | 項目 | 嚴重 | 位置 | 決議 |
|---|---|---|---|---|
| 1 | 4 處 `DispatchQueue.main.asyncAfter` 用作 UI 延遲 → `Task { try? await Task.sleep(for: .seconds(X)); ... }` | 🔵 | `EditAttractionView:151`、`EditRestaurantView:148`、`EditHotelView:232`、`HotelAnnouncementView:39`（`PlacePhotoView:171` 是 UIKit context 不算） | 🔗 holding box |
| 2 | 2 處 `UIApplication.shared.connectedScenes` 找 rootVC 來 present `UIActivityViewController` → 改用既有 `Components/ShareSheet.swift` wrapper 或 `ShareLink` | 🔵 | `ExpenseExportView:283`、`DocumentListView:154` | 🔗 holding box |
| 3 | 19 處 `DateFormatter()` 直接建立 → iOS 15+ `Date.FormatStyle`（`date.formatted(date:time:)` / `.formatted(.dateTime.month().day())`） | 🔵 | 跨多檔（Journal / Member / Team / Finance / Output） | 🔗 holding box |
| 4 | ~9 處 `.onAppear { loadXxx() }` 內含 async 邏輯 → 改 `.task { ... }`（自動 cancel-on-disappear） | 🔵 | 18 個 `.onAppear` 待動工時逐個確認哪些真有 async；現有 4 處已用 `.task` | 🔗 holding box，動工時先深讀確認範圍 |

**已撤回**（grep 看似命中但其實不是 issue）：

- `String(format: "%02d", ...)` 5 處：補零最 idiomatic 寫法
- `ForEach(items, id: \.self)` 20+ 處：多數是 `String` / `Int` / `enum`，不是 `Identifiable`，`id: \.self` 是必要寫法

**未掃到的盲點**（須讀程式碼才能判斷，這次跳過）：金額顯示用 `String(format:)`、`@State` 存衍生資料、動畫 / `withAnimation` 寫法、`ZStack` / `GeometryReader` 手動 layout 是否可改 `Layout` protocol。

### Batch 4：App / Extensions / Utilities（已掃，2026/04/29）

7 檔，最乾淨的一批。`@AppStorage` / `Schema([...])` / `ModelConfiguration` / Swift 5.9 expression switch 等現代寫法都已採用。

| # | 項目 | 嚴重 | 位置 | 決議 |
|---|---|---|---|---|
| 1 | `DebugDataGenerator` 兩處冗餘 `await MainActor.run { ... }` 包覆（struct 已標 `@MainActor`，async function 預設在 main actor 跑，不必再 wrap）。延續 Build 14 既有清理精神 | 🔵 | `DebugDataGenerator.swift:48, 64` | 🔗 holding box |

**未列入**（純優化，使用者感不到）：
- `ContentView.swift:25` TabView 用舊 `tabItem` 寫法 → iOS 18 新 `Tab` API：5 個固定底部 tab 視覺零差別，未來要 sidebar 才有意義
- `Color(hex:)` init 解析失敗 fallback 黑色 → `init?` failable：純風格

四批掃完，整體**現代化程度比預期高**：

**Holding box（待全局決策一次處理）**：

- Managers #2a, #2b, #3, #4：CalendarManager async 化、NetworkMonitor 改 Task、7 singleton 標 `final`、strict concurrency 警告
